# 切断工程「材料使用数」统计与反映方案

> 在切断・面取指示页面按产品统计「使用材料数」（以管理コード去重），支持今日/翌日/指定日查看，并通过「使用数反映」写入材料使用済表并更新 material_stock.planned_usage。

---

## 1. 需求与口径

### 1.1 统计对象与业务前提

- **统计对象**：切断工程（cutting_management）中，**每个产品**的「使用材料数」。
- **业务前提**：一产品对应一种材料（一产品一材料）。
- **去重依据**：用**管理コード**（management_code）统计——每条切断指示对应一个管理コード，统计时按管理コード去重，避免重复计数。

### 1.2 指标含义

- **每个产品的使用材料数** = 该产品在当日（及当前筛选条件下）的**不重复管理コード数量**（即该产品被多少条切断指示使用）。
- **当日整体使用材料数** = 当日所有切断指示的**不重复管理コード总数**（与「当日切断指示条数」一致）。

### 1.3 数据来源

- 现有 cutting_management list API 返回的「今日」「翌日」列表，每条含 `product_cd`、`product_name`、`management_code`。
- **去重键**：management_code。**分组维度**：产品（product_cd）。

---

## 2. 统计设计

### 2.1 按产品汇总

- **维度**：产品（product_cd / product_name）。
- **指标**：该产品在当日列表中的「使用材料数」= 该产品下 **distinct management_code** 的个数。
- **计算**：对当日列表按 product_cd 分组，每组内对 management_code 去重计数。

### 2.2 当日合计

- **当日使用材料数合计** = 当前列表中 **distinct management_code** 的总数。

### 2.3 空值处理

- **management_code 为空**：该行不参与统计（建议不计入）。
- **product_cd 为空**：可归为「未设定产品」单独一行，或排除。

### 2.4 今日与翌日联合去重（同一管理コード只算一次）

**业务含义**：同一管理コード若既在今日又在翌日（例如今日没加工完、复制到翌日继续），表示同一条指示、同一批材料，今日已用该材料，翌日仍在用「昨天的材料」→ **两天同一管理コード只算一次**。

**统计规则**：

- **今日的使用材料数**：以**今日列表**为基准，统计今日列表中 **distinct management_code** 的数量（含在翌日也出现的 code，一律算在「今日」）。
- **翌日的使用材料数**：只统计**翌日列表中、且未在今日列表中出现的** management_code（翌日「新增」的指示/材料）。今日已有的 management_code 翌日再出现则**不再计入翌日**。

**按产品汇总**：

- 今日：按产品分组，每组内对今日列表的 management_code 去重计数。
- 翌日：按产品分组，每组内只对「翌日列表中且不在今日列表中的」management_code 去重计数。

**多日范围汇总（将来）**：同一 management_code 在统计范围内只计 1 次（跨日去重）。

---

## 3. 展示方案

### 3.1 今日 / 翌日

- **位置**：在「切断指示-今日」「切断指示-翌日」区块内、表格下方增加**按产品汇总表**。
- **内容**：
  - 表头：製品CD、製品名、使用材料数（管理コード数）。
  - 每行：一个产品 + 该产品当日的不重复管理コード数。
  - 合计行：当日使用材料数合计。
- **翌日**：数据为翌日列表中且未在今日列表中出现的 management_code（翌日「新增」），与 2.4 规则一致。
- **筛选**：汇总基于当前已加载的今日+翌日列表（生产日、切断机、生产月筛选已生效）。

### 3.2 指定日期查看所需材料数

- **功能**：用户可选择**任意日期**，查看该日「所需材料数」（与今日/翌日相同口径：按产品、不重复管理コード数；可按材料汇总展示）。
- **展示方式**（二选一或并存）：
  - (a) 切断指示区块内增加「指定日」区域：日期选择器 + 该日的按产品汇总表与合计；
  - (b) 单独弹窗/抽屉「指定日の材料数」：内嵌日期选择器，选择后请求该日数据并展示。
- **数据来源**：调用 cutting_management list API，传 `production_day=指定日`（及可选切断机），取得该日列表后在前端按产品（及可选按材料）做 distinct management_code 汇总。
- **仅查看**：不写入材料使用済表、不更新 material_stock。

---

## 4. 数据管理

### 4.1 谁存、谁算

- **画面用统计**：实时从 cutting_management 列表计算，不落库，用于按产品汇总表展示。
- **持久化**：统计出的材料使用数写入**材料使用済表**，并据此**自动更新 material_stock.planned_usage**。
- **数据权威**：cutting_management 为业务源；材料使用済表为使用数持久记录；material_stock.planned_usage 由材料使用済表驱动更新。

### 4.2 业务使用场景

- **日常**：在切断指示页查看今日/翌日各产品使用材料数，便于排产与材料准备。
- **指定日**：通过指定日期查看任意一日所需材料数，仅查看不写入。
- **筛选**：切换生产日、切断机时，列表重载，汇总表自动更新。

---

## 5. 材料使用済表与写入流程

### 5.1 材料使用済表

- **目的**：落库统计结果，便于追溯，并作为更新 material_stock.planned_usage 的依据。
- **表名建议**：`material_usage_record`（材料使用実績）或 `cutting_material_usage`。
- **建议结构**（按材料 + 日一条）：
  - `id`（主键）
  - `usage_date`（date）— 使用日（生产日）
  - `material_cd`（varchar）— 材料CD
  - `material_name`（varchar）— 材料名
  - `usage_count`（int）— 使用数（该日该材料的不重复管理コード数）
  - `source`（varchar）— 来源，如 `'cutting'`
  - `created_at` / `updated_at`
  - 唯一约束：`(usage_date, material_cd, source)`，便于按日覆盖写入。

### 5.2 「使用数反映」单独按钮与确认后写入

- **触发**：「使用数反映」为**单独按钮**（与「实绩确定」等分开），仅在使用户完成确认流程后才写入。
- **流程**：
  1. 用户点击「使用数反映」→ 前端带当前「今日」生产日（及可选翌日）请求后端。
  2. 后端按 2.4 规则按**材料**汇总使用数，**仅返回待写入的使用数量列表**（不写入）。列表建议含：usage_date、material_cd、material_name、usage_count 等。
  3. 前端弹出**使用数量确认列表**（对话框或抽屉），供用户核对。
  4. 用户点击「确认」后，前端再请求后端执行**写入**：写入材料使用済表并更新 material_stock.planned_usage；用户取消则不做任何写入。
- **写入逻辑**：按 usage_date + material_cd + source 做 UPSERT。材料维度需从 cutting_management 的 product 关联 products 取得 material_cd；若无则用 material_name 对应材料主数据或暂不写入。

### 5.3 自动更新 material_stock.planned_usage

- **时机**：在写入材料使用済表之后**立即**执行（同一事务）。
- **规则**：对本次涉及的 (material_cd, usage_date)，用材料使用済表中该 (material_cd, usage_date) 的 usage_count 作为 **planned_usage** 更新 material_stock。
- **material_stock**：现有表为 `(material_cd, date)` 唯一，`planned_usage` 为 int。已存在则 UPDATE；不存在则按业务决定是否 INSERT 新行。
- **事务**：材料使用済表写入与 material_stock 更新放在同一事务中。

### 5.4 使用材料事后有修改时的处理

- **场景**：某日已执行「使用数反映」后，用户又对切断指示做了修改（增删改、变更产品/材料、顺延等），导致该日「应反映使用数」与已写入不一致。
- **处理原则**：允许对同一日**再次执行「使用数反映」**，**覆盖**该日在材料使用済表中的记录，并重新更新 material_stock.planned_usage。
- **具体做法**：
  - 再次点击「使用数反映」→ 同样先出使用数量列表 → 用户确认 → 写入时对该日+材料做 UPSERT，再按最新结果更新 material_stock。
  - 可选提示：若该日已有反映记录，可提示「该日已有反映，确认后将覆盖原数据」。
  - 某材料在该日使用数变为 0：反映时写入 usage_count=0（或删除该条），并将 material_stock 中该材料该日 planned_usage 置为 0。
- **不自动同步**：切断指示的日常增删改不自动回写材料使用済表或 material_stock，仅在用户主动「使用数反映」并确认后才更新。

## 6. 实现要点

### 6.1 前端（画面统计与展示）

- 新增计算属性（今日/翌日）：今日以 cuttingManagementList 按产品分组、组内 distinct management_code 计数；翌日先排除已在今日出现的 management_code 后再按产品分组、组内 distinct 计数。
- 在切断指示-今日/翌日表格下方增加按产品汇总表；页脚可增加「使用材料数（管理コード数）：N」。
- **指定日期**：增加日期选择器（及可选切断机），选择后请求 list API（production_day=指定日），用返回列表按产品（及可选按材料）做 distinct management_code 汇总，展示该日所需材料数；仅查看。
- 切断指示区块增加**「使用数反映」单独按钮**：点击 → 请求预览 API → 弹出使用数量确认列表 → 用户确认后请求执行写入 API；若该日已有反映可提示「该日已有反映，确认后将覆盖」。

### 6.2 后端

- **Migration**：新建材料使用済表（如 material_usage_record），含 usage_date, material_cd, material_name, usage_count, source，唯一 (usage_date, material_cd, source)。
- **两步 API**：
  - **预览/列表**：入参生产日（及可选翌日），从 cutting_management 按 2.4 规则按材料汇总，**仅返回**使用数量列表（不写入）。
  - **执行写入**：入参为用户确认后的列表（或再次用相同生产日计算），写入材料使用済表（UPSERT）并更新 material_stock.planned_usage，同一事务。同一日可多次调用（再次反映时覆盖）。
- 材料维度：cutting_management 通过 product_cd 关联 products 取 material_cd；若无则按 material_name 查 materials 或暂不写入。

## 7. 小结

| 项目 | 内容 |
|------|------|
| 统计维度 | 按产品（product_cd）；写入材料使用済表时按材料（material_cd）汇总 |
| 指标含义 | 使用材料数 = 不重复管理コード数；今日+翌日同一管理コード只计 1 次（计在今日） |
| 去重键 | management_code |
| 画面 | 今日/翌日按产品汇总表 + 页脚合计；**指定日**可选日期查看该日所需材料数（仅查看） |
| 材料使用済表 | material_usage_record（usage_date, material_cd, material_name, usage_count, source），按日按材料 UPSERT |
| material_stock | 写入使用済表后，同一事务内更新 planned_usage（material_cd + date 对应） |
| 使用数反映 | 单独按钮；先出使用数量列表供确认，确认后再写入材料使用済表并更新 planned_usage |
| 事后修改 | 使用材料有变更时，再次执行「使用数反映」并确认，覆盖该日记录并重算 planned_usage |
